# Authoring Roadmap

This is the practical staged plan for turning the repo from seeded knowledge into a playable system.

## M0 — Preserve truth

- Keep the knowledge base authoritative.
- Refuse feature work that breaks the canon or engine stance.
- Treat docs as part of implementation, not decoration.

## M1 — Minimal runtime skeleton

Build the smallest viable runtime with:

- deterministic tick loop
- process state container
- command dispatcher
- semantic event log
- journal and map projection shell
- content loader for a first-pass DSL

## M2 — First playable district

Author one compact district made of:

- one archive room chain
- one growth bed
- one looping path system
- one oracle bargain
- one mirror surface
- one local engine node

Goal:
Prove the game's central loop before scaling content breadth.

## M3 — Representation as play

Add the player-facing instruments that make the game special:

- semantic map
- journal and note surfaces
- replay trace view
- thread list and commitment panel
- contradiction and debt surfacing

## M4 — Progression and contradiction service

Implement the runtime service that controls:

- trust to acceptance phase movement
- family availability
- contradiction timing
- self-reference escalation
- lawful failure pacing

## M5 — Content expansion

Expand horizontally by authoring more situations with the same stable primitives.

Do not expand by hardcoding exceptions.

## M6 — Tooling and quality

Build the authoring and debugging infrastructure:

- content validation
- lints for canonical drift
- replay debugger
- migration tests
- content-pack versioning
- saved-world compatibility checks

## Working rule

Whenever a milestone feels blocked, check whether the project is missing a semantic abstraction or only missing more content. In The Pit, missing abstractions are usually the real blocker.
