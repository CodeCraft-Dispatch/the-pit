# Content Authoring

Read these first:

- `docs/knowledge-base/puzzle-families.md`
- `docs/knowledge-base/content-system.md`

## Authoring rule

Every content beat should define:

- family
- arc pressure
- visible surface
- available costly verbs
- hidden process behavior
- possible closure states
- contradiction opportunity
- downstream consequence

## Working method

1. choose the family and the current progression pressure
2. define what the player sees first
3. define what commitment costs
4. define what counts as apparent progress and meaningful progress
5. define how the beat closes or mutates
6. define how the beat affects another thread

## Avoid

- one-off content that cannot be generalized
- opaque puzzles with no readable process state
- contradictions that feel like gotchas instead of lawful reversals
- content that ignores the map, journal, or replay surfaces
